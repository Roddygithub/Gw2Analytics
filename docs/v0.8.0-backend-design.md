# v0.8.0 backend design — Webhook subscriptions for parse-completion

**Status:** Draft (post-v0.7.0)
**Web dependency:** none (the v0.8.0 backend is independently
useful; the v0.8.0 web layer consumes one of the new endpoints for
a "subscription status" panel — see §6 below)
**Backend target:** v0.8.0

## 1. Motivation

The v0.7.0 backend exposes 4 new endpoints that surface the
player-centric + per-fight roll-ups. The ingestion flow is still
synchronous from the analyst's perspective: the analyst POSTs a
combat log to `/api/v1/uploads`, polls `/api/v1/fights` until the
fight shows up, then drills in. This poll-until-done pattern is
fine for a human in the loop, but it's a poor fit for:

- **Third-party bots** (e.g. guild-notification, raid-end pings)
  that want to post a "parse complete" notification to a channel
  when a raid ends.
- **CI integrations** that want to fail a build if a new parse
  is missing expected fights.
- **Dashboards** that want to display a "live" upload list without
  having to re-fetch every N seconds.

The v0.8.0 cycle adds **webhook subscriptions**: the analyst (or an
external integrator) registers a URL + filter triple, the gateway
fires an HTTP POST to the URL when a matching parse completes, and
the integrator handles the notification out-of-band. The
integrator can then fetch the new fight via the existing
`GET /api/v1/fights/{id}` endpoint.

## 2. Scope

### 2.1 In scope

- **3 new endpoints:**
  - `POST /api/v1/webhooks` — register a new subscription.
  - `GET /api/v1/webhooks` — list subscriptions for the requesting
    user.
  - `DELETE /api/v1/webhooks/{id}` — revoke a subscription.
- **1 new Postgres table** (`webhook_subscriptions`).
- **1 new background worker** that polls the `upload_completed`
  log table + fires the matching webhooks.
- **Retry + dead-letter queue (DLQ)**: 3 retries with exponential
  backoff (1s, 10s, 100s) before the webhook is moved to a DLQ
  table for manual inspection.
- **HMAC-SHA256 signature header** (`X-Gw2Analytics-Signature`)
  on every outbound POST so the receiver can verify the request
  is genuinely from the gateway.
- **Tests:**
  - 8 unit tests for the HMAC signing helper.
  - 6 integration tests for the 3 endpoints (CRUD + auth).
  - 4 worker tests (success, retry, DLQ, filter mismatch).

### 2.2 Out of scope

- **GraphQL subscription channel**. Webhook POST is the canonical
  surface for v0.8.0; GraphQL is a v0.9.0+ enhancement.
- **Multi-tenant scoping**. v0.8.0 is single-tenant (one
  deployment, one auth scope). Multi-tenant is a v1.0+ concern.
- **Webhook replay / redelivery UI**. The DLQ table is the
  canonical "replay this" surface; a UI for it is a v0.9.0
  enhancement.
- **Per-event-kind filtering** (e.g. "only fire on
  `BuffRemovalEvent`"). The v0.8.0 filter is coarse (per-upload
  or per-encounter); fine-grained filtering is a v0.9.0
  enhancement.

## 3. API contract

### 3.1 `POST /api/v1/webhooks`

**Request body:**
```json
{
  "url": "https://example.com/webhook",
  "filter": {
    "kind": "upload_completed"
  },
  "description": "Out-of-band parse-completion notifications"
}
```

**Response (201):**
```json
{
  "id": "wh_abc123",
  "url": "https://example.com/webhook",
  "filter": { "kind": "upload_completed" },
  "description": "Out-of-band parse-completion notifications",
  "secret": "whsec_xxxxxxxxxxxxxxxxxxxx",
  "created_at": "2026-07-07T12:00:00+00:00"
}
```

The `secret` is returned ONCE (in the 201 response) and never
again. The integrator must store it securely; it's used to verify
the `X-Gw2Analytics-Signature` header on every outbound POST.

### 3.2 `GET /api/v1/webhooks`

**Response (200):** array of subscription summaries (no `secret`).

### 3.3 `DELETE /api/v1/webhooks/{id}`

**Response (204):** no body.

### 3.4 Outbound POST (gateway → subscriber)

```
POST /webhook HTTP/1.1
Host: example.com
Content-Type: application/json
X-Gw2Analytics-Signature: sha256=abc123...
X-Gw2Analytics-Delivery: dly_xyz789
User-Agent: Gw2Analytics-Webhook/0.8.0

{
  "kind": "upload_completed",
  "upload_id": "upl_abc",
  "fight_id": "fix_001",
  "sha256": "...",
  "started_at": "2026-07-07T12:00:00+00:00"
}
```

The receiver must verify `X-Gw2Analytics-Signature` against the
stored `secret` (HMAC-SHA256 of the raw body) before trusting the
payload.

## 4. Database schema

```sql
CREATE TABLE webhook_subscriptions (
  id TEXT PRIMARY KEY,                       -- wh_<uuid>
  url TEXT NOT NULL,
  filter JSONB NOT NULL,                     -- { kind: "upload_completed", ... }
  description TEXT,
  secret TEXT NOT NULL,                      -- whsec_<32 bytes base64>
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at TIMESTAMPTZ                      -- NULL = active
);

CREATE TABLE webhook_deliveries (
  id TEXT PRIMARY KEY,                       -- dly_<uuid>
  subscription_id TEXT NOT NULL REFERENCES webhook_subscriptions(id),
  upload_id TEXT NOT NULL,
  attempt INT NOT NULL,                      -- 1, 2, 3
  status_code INT,                           -- HTTP status from the receiver
  error TEXT,                                -- network error if status_code is NULL
  delivered_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE webhook_dlq (
  id TEXT PRIMARY KEY,
  subscription_id TEXT NOT NULL,
  upload_id TEXT NOT NULL,
  payload JSONB NOT NULL,
  last_error TEXT,
  moved_to_dlq_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 5. Worker design

A new `WebhookDispatcherWorker` runs in the gateway's background
process pool. It:

1. Polls the `webhook_deliveries` table for rows with
   `attempt < 3` and `delivered_at IS NULL` (every 5s).
2. For each row, looks up the subscription, builds the outbound
   payload, signs it with HMAC-SHA256, and POSTs to the
   subscription URL with a 10s timeout.
3. On 2xx response: marks the row as delivered.
4. On non-2xx / network error: increments `attempt` and
   `delivered_at = now() + exponential_backoff(attempt)`.
5. On `attempt == 3` failure: moves the row to `webhook_dlq`.

The 5s poll interval is the latency floor. For lower latency,
switch to a Postgres `LISTEN/NOTIFY` channel (v0.9.0).

## 6. Future work (v0.9.0+)

- **v0.8.0 web consumption**: a small "Webhooks" tab in the v0.8.0
  web layer (a `/webhooks` page) that lets the analyst register +
  list + revoke subscriptions. Strict parallel of the v0.7.0
  `/account` page.
- **GraphQL subscription channel**: for dashboards that want
  push-based updates without polling the gateway's REST surface.
- **Per-event-kind filtering**: a `filter.events: ["BuffRemoval"]`
  shape that lets the integrator narrow the delivery to specific
  event kinds (requires extending the v0.7.0 events blob to carry
  a per-event kind tag).
- **Webhook replay / redelivery UI**: a panel on the `/webhooks`
  page that lists the DLQ rows and lets the analyst click "Replay"
  to re-fire a specific delivery.

## 7. Open questions

1. **Auth model**: the v0.8.0 webhooks are scoped to a single
   "user" (the deployment owner). A multi-user deployment needs a
   per-user scope; deferred to v1.0.
2. **Secret rotation**: the v0.8.0 secret is set once at creation
   and never rotated. A `POST /webhooks/{id}/rotate-secret` endpoint
   is a v0.9.0 enhancement.
3. **HTTPS-only**: should the gateway refuse to register a
   non-HTTPS URL? Default to "yes, refuse HTTP" (the only exception
   is `http://localhost` for local development).
