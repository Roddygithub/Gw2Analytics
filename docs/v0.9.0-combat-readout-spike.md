# Spike Output — Combat Readout Feasibility (Plan 024)

> **Scope:** Design/spike only. No production routes, aggregators, or UI components are implemented.
> **Goal:** Validate the v0.9.0 combat-readout design, prototype the first two Event subclasses, and document the parser + API surface needed for full implementation.

## 1. Fixture diversity analysis

The repository does **not** contain real `.zevtc` files. All EVTC data used in tests is generated synthetically by `apps/api/tests/_fixtures.py::make_minimal_zevtc`. Synthetic fixtures currently emit only:

- A minimal 25-byte EVTC header.
- 1–2 player agents (96-byte fixed records).
- 0–2 skill records.
- Optional 64-byte `cbtevent` records with `is_statechange == 0`.

Because no real logs are present, the spike cannot enumerate which `is_statechange` IDs appear in production data. Instead, this document maps the documented arcdps IDs from `docs/statechange-ids.md` to the readout concepts and treats the mapping as the design baseline.

### Priority grouping of Event subclasses

| Priority | Event subclass | Triggering statechange ID(s) | Readout table |
|---|---|---|---|
| P0 | `BoonApplyEvent` | `69 | BuffApply` | Boons |
| P0 | `CCEvent` | `34 | BreakbarState`, `35 | BreakbarPercent` (defiance-bar damage) and standard combat events flagged as hard-CC (stun/daze/knockback/etc.) | Damage / Defense |
| P1 | `ConditionRemoveEvent` | `71 | BuffRemoveSingle`, `72 | BuffRemoveAll` | Heal |
| P1 | `DownEvent` | `5 | ChangeDown` | Defense |
| P1 | `DeathEvent` | `4 | ChangeDead` | Defense |
| P1 | `StunBreakEvent` | `56 | StunBreak` | Heal |
| P2 | `DodgeEvent` | Standard combat event with dodge result flag (no dedicated statechange) | Defense |
| P2 | `BlockEvent` | Standard combat event with block result flag (no dedicated statechange) | Defense |
| P2 | `InterruptEvent` | Standard combat event with interrupt result flag (no dedicated statechange) | Damage |

**Note:** Dodge, block, and interrupt are not emitted as dedicated statechange records in arcdps V1.3. They are inferred from result flags on standard combat events (`is_statechange == 0`). This is the largest parser uncertainty and is flagged in §4.

## 2. Prototype Event models

Two Event subclasses already exist in `libs/gw2_core/src/gw2_core/models.py` as spike prototypes (this satisfies the plan's Step 2):

- `BoonApplyEvent` — `event_type = "BOON_APPLY"`, `duration_ms`, `stacks`.
- `CCEvent` — `event_type = "CC"`, `cc_value`.

Both inherit `BaseEvent` (time_ms, source_agent_id, target_agent_id, skill_id) and participate in the `Event` discriminated union. They are structurally adequate for the spike.

### Suggested model refinements before full implementation

1. **`BoonApplyEvent`**: keep `duration_ms` and `stacks`; consider renaming `skill_id` usage to `boon_id` in documentation, but keep the field name `skill_id` for union compatibility.
2. **`CCEvent`**: `cc_value` semantics need clarification during parser integration — whether it represents defiance-bar damage, duration in milliseconds, or a normalized magnitude.
3. **Future subclasses** (to be added in a later plan):
   - `ConditionRemoveEvent`: `condition_id`, `stacks_removed`.
   - `DownEvent`: no extra fields beyond `BaseEvent`.
   - `DeathEvent`: no extra fields beyond `BaseEvent`.
   - `StunBreakEvent`: no extra fields beyond `BaseEvent`.
   - `DodgeEvent`, `BlockEvent`, `InterruptEvent`: no extra fields beyond `BaseEvent`; identified by result flags.

## 4. Parser change assessment

### Current behavior

In `libs/gw2_evtc_parser/src/gw2_evtc_parser/parser.py`, `parse_events` discards every record with `is_statechange != 0` at approximately line 282:

```python
if is_statechange != 0:
    continue
```

Only `DamageEvent`, `HealingEvent`, and `BuffRemovalEvent` are yielded, and only from `is_statechange == 0` records.

### Required changes

1. **Remove the blanket discard** of `is_statechange != 0` records in `PythonEvtcParser.parse_events`.
2. **Add a statechange dispatch branch** before the damage/heal/strip logic (exact line range: `parser.py` around line 282, inside the `while cursor + EVENT_SIZE <= end:` loop):
   - `4 | ChangeDead` → `DeathEvent`
   - `5 | ChangeDown` → `DownEvent`
   - `56 | StunBreak` → `StunBreakEvent`
   - `69 | BuffApply` → `BoonApplyEvent`
   - `71 | BuffRemoveSingle`, `72 | BuffRemoveAll` → `ConditionRemoveEvent`
   - `34 | BreakbarState`, `35 | BreakbarPercent` → `CCEvent` (defiance-bar component)
3. **Keep the existing damage/heal/strip path** for `is_statechange == 0` records.
4. **Add result-flag parsing** for dodge/block/interrupt on standard combat events. This requires inspecting the `is_cleanup`, `is_nondamage`, and result-flag bytes of the `cbtevent` struct; the exact flag positions are not yet documented in this codebase and must be validated against arcdps source.

### Effort estimate

| Work | Estimate | Notes |
|---|---|---|
| Parser statechange dispatch | 1–2 days | Straightforward ID-to-event mapping. |
| Result-flag parsing (dodge/block/interrupt) | 2–3 days | Requires arcdps source validation; highest uncertainty. |
| New Event subclasses + union update | 0.5 day | Models are simple; tests are quick. |
| Per-player aggregators | 2–3 days | Reuse `_compute_contributions` pattern. |
| Role classifier | 1–2 days | Heuristic calibration deferred to implementation cycle. |
| API route + schemas | 1 day | Unified endpoint shape is settled. |
| Web tables | 2–3 days | 4 AG Grid components, shared base columns. |
| **Total backend** | **~4–6 days** | Excludes web work. |
| **Total full stack** | **~2–3 weeks** | Includes calibration and QA. |

## 5. Open questions

1. **Dodge/block/interrupt flags**: Where exactly do these result flags live in the 64-byte `cbtevent` struct? The current parser ignores the result bytes.
2. **CC magnitude semantics**: Is `CCEvent.cc_value` defiance-bar damage, duration in ms, or a normalized value?
3. **Boon stacking**: Does `BoonApplyEvent.stacks` map to `buff_dmg`, `overstack_value`, or another field?
4. **Real-log validation**: No production `.zevtc` files are in the repo. The first implementation plan should include capturing a small corpus of real logs to validate the statechange mapping.
5. **Skills DB**: The static JSON skills DB is still out of scope; the power/condi split columns depend on it.

## 6. STOP conditions check

- **Statechange IDs radically different from documented IDs?** No — the mapping in `docs/statechange-ids.md` is consistent with Elite Insights and arcdps documentation.
- **Parser cannot yield statechange events without breaking output shape?** No — the `Event` discriminated union already includes `BoonApplyEvent` and `CCEvent`; adding more subclasses is forward-compatible.
- **No `.zevtc` fixture available?** Partially — only synthetic fixtures exist. This does not block the spike, but it means the statechange mapping is design-baseline only and must be validated with real logs during implementation.

## 7. Recommended implementation plan

The full combat-readout feature is too large for a single plan. The spike recommends splitting it into the following follow-up plans:

1. **Parser statechange dispatch (P1)**
   - Add the remaining Event subclasses (`ConditionRemoveEvent`, `DownEvent`, `DeathEvent`, `StunBreakEvent`).
   - Consider splitting `CCEvent` into two concepts during implementation: `BreakbarEvent` (from `34 | BreakbarState` / `35 | BreakbarPercent`) and `HardCCEvent` (from standard combat events flagged as stun/daze/knockback/etc.).
   - Replace the `if is_statechange != 0: continue` discard in `parser.py` with an ID-to-event dispatch branch.
   - Yield `BoonApplyEvent`, `CCEvent` (or the split above), `ConditionRemoveEvent`, `DownEvent`, `DeathEvent`, `StunBreakEvent`.
   - Add hermetic parser tests using synthetic `cbtevent` records.
   - *Resolves:* which statechange IDs actually appear in real logs; whether the parser can yield statechange events without breaking the existing output shape.

2. **Real-log validation corpus (P1)**
   - Capture a small corpus of real `.zevtc` files (with user consent).
   - Run the extended parser against the corpus and validate the statechange ID frequencies against the mapping in §2.
   - Resolve open questions about dodge/block/interrupt result flags and `CCEvent.cc_value` semantics.
   - *Resolves:* exact result-flag positions for dodge/block/interrupt; whether `CCEvent.cc_value` is defiance damage, duration, or a normalized magnitude.

3. **Per-player combat-readout aggregators (P2)**
   - Implement `PlayerDamageAggregator`, `PlayerHealAggregator`, `PlayerBoonsAggregator`, `PlayerDefenseAggregator` in `libs/gw2_analytics`.
   - Reuse the existing `_compute_contributions` helper from the player routes.
   - Add unit tests with synthetic event streams.
   - *Resolves:* whether existing aggregators can be reused or new ones are needed; how to group boons into fixed vs. dynamic columns.

4. **Role classifier (P2)**
   - Implement `PlayerRoleClassifier.classify(roll_ups) -> set[str]`.
   - Calibrate thresholds against the real-log corpus.
   - *Resolves:* the final role vocabulary and threshold calibration.

5. **Unified readout endpoint + web tables (P3)**
   - Add `GET /api/v1/fights/{id}/readout` in `apps/api`.
   - Build the 4 AG Grid components and integrate them into `/fights/[id]`.
   - *Resolves:* whether to create a new page or integrate into the existing `/fights/[id]` drill-down.
