# v0.8.0 web design — Account-level historical timelines

**Status:** Draft (post-v0.7.1)
**Backend dependency:** v0.7.0 (already shipped)
**Web target:** v0.8.0

## 1. Motivation

The v0.7.1 web layer ships a player-centric surface (2 new pages + 5
new components + 4 new fetchers + 13 new tests). The `/players` page
shows a cross-fight roll-up per account; the `/players/[account_name]`
page shows the per-fight breakdown for one account. Both views are
**point-in-time**: the analyst sees the cumulative totals, but has no
way to see how the player has trended over time.

The v0.8.0 cycle adds the missing **time dimension**: a
historical-timeline panel on the `/players/[account_name]` page that
charts the player's `total_damage` / `total_healing` /
`total_buff_removal` over the most-recent N fights (default N=20, with
a "Load more" affordance to paginate deeper). The chart is rendered
inline (no external charting library, matching the v0.7.1
`EventWindowsChart` precedent) so the bundle stays slim.

## 2. Scope

### 2.1 In scope

- **New endpoint client**: `fetchPlayerTimeline(accountName, opts)` mirroring
  `GET /api/v1/players/{account_name}/timeline?limit=N&offset=M` (a
  new v0.7.1-api endpoint; the v0.8.0 backend ships this in the same
  release).
- **New component**: `<PlayerTimelineChart />` — inline SVG line chart
  (3 lines, one per metric, shared x-axis, hover tooltip showing the
  exact values for the bucket under the cursor). Mirrors the visual
  + behavioural contract of `EventWindowsChart` (side-by-side bars).
- **New component**: `<PlayerTimelineLegend />` — small legend strip
  (3 colour swatches + label) for the 3 metrics.
- **Modified page**: `/players/[account_name]/page.tsx` — adds a new
  section between the cross-fight stat cards and the per-fight
  breakdown, titled "Recent timeline (last N fights)".
- **New tests**:
  - 3 page-level cases (timeline renders, 404 path, "Load more" path)
  - 2 component-level cases (hover tooltip, empty timeline)
  - 1 E2E case (golden path: profile → timeline → hover)

### 2.2 Out of scope

- Real-time DPS meter (a separate, larger feature; see §6 below).
- Cross-account comparison ("compare me to my raid-mate"). The
  per-account timeline is enough for v0.8.0; cross-account
  comparison is a candidate for v0.9.0.
- Mobile-first redesign. The desktop layout is the canonical
  analyst surface; mobile is a future enhancement.
- Exporting the timeline as PNG / SVG. CSV export is already
  covered by the v0.7.1 `CsvDownloadButton` (the timeline section
  reuses it with a `${account_name}-timeline.csv` filename).

## 3. API contract

### 3.1 `GET /api/v1/players/{account_name:path}/timeline`

**Query params:**
- `limit` (int, default 20, range [1, 100]): page size.
- `offset` (int, default 0, range [0, ∞)): page offset.

**Response (200, `application/json`):**
```json
{
  "account_name": "TestAccount.1234",
  "total": 42,
  "limit": 20,
  "offset": 0,
  "points": [
    {
      "fight_id": "fixture-fight-002",
      "started_at": "2026-07-07T13:30:00+00:00",
      "total_damage": 0,
      "total_healing": 0,
      "total_buff_removal": 0
    },
    {
      "fight_id": "fixture-fight-001",
      "started_at": "2026-07-07T12:00:00+00:00",
      "total_damage": 85000,
      "total_healing": 12000,
      "total_buff_removal": 50
    }
  ]
}
```

**Errors:**
- `404` (player not found): mirror of the existing
  `GET /api/v1/players/{account_name}` 404 contract.
- `422` (out-of-range limit/offset): mirror of the existing
  v0.7.0 query-param validation contract.

**Sort order:** points are ordered by `started_at DESC` (most-recent
first), with `fight_id ASC` as the tiebreaker for fights that share a
`started_at` (the v0.7.0 aggregator's deterministic-ordering contract).

### 3.2 New fetcher helper

`web/src/lib/api.ts` adds:

```ts
export interface PlayerTimelinePoint {
  fight_id: string;
  started_at: string;
  total_damage: number;
  total_healing: number;
  total_buff_removal: number;
}

export interface PlayerTimeline {
  account_name: string;
  total: number;
  limit: number;
  offset: number;
  points: PlayerTimelinePoint[];
}

export async function fetchPlayerTimeline(
  accountName: string,
  opts: { limit?: number; offset?: number } = {},
): Promise<PlayerTimeline>;
```

## 4. Component design

### 4.1 `PlayerTimelineChart`

Strict parallel of `EventWindowsChart` (inline SVG, no external
charting lib). Differences:
- 3 polylines instead of 2 sets of bars.
- Hover tooltip rendered as an absolutely-positioned `<div>` overlay.
- x-axis labels are the `started_at` timestamps (formatted as
  `MM/DD HH:MM`, locale-aware via `Intl.DateTimeFormat`).
- y-axis labels are the 3 metric values (formatted via the
  `Intl.NumberFormat` compact notation, e.g. `12.5K`).
- y-axis is scaled to the MAX of the 3 metrics (so all 3 lines are
  visible in the same frame).

### 4.2 `PlayerTimelineLegend`

3 colour swatches (red, green, orange) with labels: "Damage",
"Healing", "Buff removal". Renders above the chart as a
right-aligned flex row.

### 4.3 Page-level wiring

```tsx
// /players/[account_name]/page.tsx
const [timelineLimit, setTimelineLimit] = useState(20);
const timeline = await fetchPlayerTimeline(account_name, { limit: timelineLimit });

// Inside the page render, between the stat cards and the breakdown:
<section>
  <h2>Recent timeline (last {timeline.points.length} fights)</h2>
  <PlayerTimelineLegend />
  <PlayerTimelineChart points={timeline.points} />
  {timeline.points.length < timeline.total ? (
    <button onClick={() => setTimelineLimit((n) => n + 20)}>
      Load more
    </button>
  ) : null}
</section>
```

The "Load more" button increments the limit (a simple "show 20 more"
pattern; offset-based pagination is a v0.9.0 enhancement).

## 5. Risks

1. **SVG performance**: with 100 points × 3 lines, the SVG has 300
   `<circle>` hover targets. Browser perf is fine for 100 points; if
   v0.8.0 is ever asked to support 1000+ points, we should switch to
   canvas (v0.9.0).
2. **Hover tooltip positioning**: the tooltip needs to track the
   mouse position accurately. Use `useRef` + `getBoundingClientRect`
   to position relative to the SVG container.
3. **Per-fight 0-0-0 rows** (a parse that yielded 0 events): the
   chart needs to render the row without crashing the line (treat as
   a data point with all-zero values, no special-case branch).

## 6. Future work (v0.9.0+)

- **Real-time DPS meter**: WebSocket-based live DPS display during a
  live parse. Significant scope (auth, reconnect, partial parse
  handling); appropriate for a dedicated v0.9.0 cycle.
- **Cross-account comparison**: pick 2-4 accounts, overlay their
  timelines on the same chart. Builds on the v0.8.0 chart component.
- **Skill build analyser**: parse the player's loadout from the
  EVTC header, show per-skill DPS contribution. Builds on the
  v0.7.1 `SkillUsageTable`.

## 7. Open questions

1. **Y-axis scale**: linear vs log? Linear is the default (matches
   the v0.7.1 `EventWindowsChart`); log is a future enhancement.
2. **Time bucketing**: per-fight (the default) vs per-day (aggregate
   all fights in a day)? Per-fight is the analyst's typical
   granularity; per-day is a v0.9.0 enhancement.
3. **"Load more" vs infinite scroll**: the button is more
   accessibility-friendly; infinite scroll is more
   mobile-friendly. Default to the button.
