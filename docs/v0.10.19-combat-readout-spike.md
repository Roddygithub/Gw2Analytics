# v0.10.19 Combat readout (F17) sizing spike

> **Companion doc:** the canonical design is at [`docs/v0.9.0-combat-readout-design.md`](./v0.9.0-combat-readout-design.md). This spike doc ENUMERATES the sub-blocks + estimates effort per sub-block + proposes the v0.10.21+ delivery order. The spike does NOT propose scope cuts; the F17 design's 4 tables + 5 shared columns ship as specced.

**Status:** Spike (post-v0.10.18.1; pre-F17-impl).
**Authoring context:** authored during the v0.10.19 mimo-half plan-landing cycle (alongside `plans/RELEASE-v0.10.19.md`); the design doc itself was authored in v0.9.0 and is not modified here.
**Target cycle(s):** v0.10.21+ (XL+ effort spans 2-3 versions per the v0.9.0 design's "Build sequence" §9).

---

## 1. Why this spike

F17 (Combat readout) is the longest-cycle + highest-analyst-value item on `docs/ROADMAP.md` §1 v1.0 candidates. It has been on the shortlist since v0.8.0 (the first design doc commit), and the v0.9.0 design doc was ratified in v0.9.0's release notes. The cycle planning has historically deferred F17 ("blocked on statechange parser + skills DB"), but without a sizing document the blockers remain haemoglobin-grade vague.

This spike locks the blockers into specific:

1. **Sub-blocks** (with files + LoC estimates)
2. **Waterfall dependencies** (which sub-block ships before which)
3. **Effort estimates per sub-block** (XS / S / M / L / XL relative)
4. **Delivery order** (the v0.10.21 + v0.10.22 + v0.10.23 cycle topology)
5. **Cut list** (what would be deferred if v0.10.21 budget is exceeded)

The spike is READ-ONLY on the design doc; the design doc remains the source of truth for the 4 tables + 5 shared columns + 10 §3.1 role terminology. The spike AUTHORITATIVELY spec's the impl-order + estimate.

## 2. Blockers — enumerated

### Blocker A — Statechange parser extension (the upstream filter gap)

**Location:** `libs/gw2_evtc_parser/src/gw2_evtc_parser/parser.py:439` — `if is_statechange != 0: continue` is the existing upstream filter that SKIPS statechange records before the REMOVE/APPLY predicates run.

**Current state:** Phase 9 step 2 (REMOVE channel) + step 3 (APPLY channel) shipped in v0.10.11. Both yield `BoonApplyEvent` records from the NON-statechange path only. The statechange-driven path (CBTS_BUFFAPPLY + CBTS_BUFFREMOVE statechange markers + defiance-bar / breakbar / down / death / dodge / block / interrupt statechange events) is UNCONSUMED.

**Why F17 needs it:** the 4 tables of the Combat readout include columns that derive ONLY from statechange records (per the §3.1 role classifier + §3 Table 1 Strips / Table 3 Boons / Table 4 Defense sub-columns). Specifically:

- Table 1 / Damage / Strips (`BuffRemovalEvent` from statechange drives)
- Table 1 / Damage / CC appliqués (statechange count for stun / daze / knockback / pull / launch / float / sink)
- Table 1 / Damage / Down contribution DPS (statechange `target → downed` transition tracking)
- Table 1 / Damage / Kills (statechange `actor → dead` transitions)
- Table 2 / Heal / Cleanses (statechange `condition removed` events)
- Table 2 / Heal / Breakstunt (statechange `stun break` events)
- Table 3 / Boons (the 6+ columns; partly derived from `BoonApplyEvent` REMOVE channel + partly from statechange `boon apply` events)
- Table 4 / Defense / CC reçus / damage reçu / deaths / dodges / blocks / interrupts (all statechange-derived)

(Future defensive state: parser will need to emit `ConditionRemoveEvent`, `CCEvent`, `DownEvent`, `DeathEvent`, `DodgeEvent`, `BlockEvent`, `InterruptEvent`, `StunBreakEvent` per the §9 Build sequence step 1's "Add 9 new Event subclasses in libs/gw2_core" list.)

**Sub-blocks (within this blocker):**

| Sub-block | Description | Effort |
|---|---|---|
| A.1 | Audit arcdps' ~150 statechange kinds (reference: Elite Insights C# `StateChange` enum) | S |
| A.2 | Map each statechange kind to the matching `Event` subclass (existing `BoonApplyEvent` REMOVE/APPLY channels cover ~30%; the remaining ~70% need new subclasses) | M |
| A.3 | Add 9 new `EventType` enum entries + 9 new Pydantic subclasses in `libs/gw2_core/src/gw2_core/models.py` (per §9 Build sequence step 1) | M |
| A.4 | Extend the parser's `cbtevent` decode loop to read the statechange kind byte (currently SKIPPED at line 439) and emit the matching subclass | XL |
| A.5 | Add 8 hermetic `parse_events` predicate-boundary tests in `libs/gw2_evtc_parser/tests/test_parser_emit_statechange.py` (one per new subclass) | M |
| A.6 | Real-fixture integration test via `test_parser_applive_realfixture.py` (extends the F1 calibration pilot to the new statechange kinds) | M |
| A.7 | Update `docs/v0.10.11-phase-9-conditions.md` + the ROADMAP §1.1 cycle shipts (Phase 9 step 4-STEPS) | S |

**Total:** XL effort (8 sub-blocks, ~1200 LoC across parser + gw2_core + tests + docs).

### Blocker B — Skills DB bootstrap (no skill-name resolution today)

**Location:** skills live as in-memory `Skill` Pydantic models (in `libs/gw2_core/src/gw2_core/models.py:181`, modeled as `id: int + name: str`) + per-fight ad-hoc via `parser._iter_skills` (`libs/gw2_evtc_parser/src/gw2_evtc_parser/parser.py:783`). **NO persistent skills catalog exists**; the parser surfaces the per-fight skills from the EVTC blob's skill table.

**Why F17 needs it:** the per-skill power/condi split (Table 1 / Damage / DPS power vs DPS condi) needs a skills catalog that classifies each `skill_id` as one of {power, condi, hybrid}. The ad-hoc per-fight skills list (returned by `parser._iter_skills`) contains only the skills used in that fight; the classification needs the OFFICIAL GW2 skill metadata (~1000 skills) loaded ONCE at server startup.

**Sub-blocks (within this blocker):**

| Sub-block | Description | Effort |
|---|---|---|
| B.1 | Decide skills DB source (the §11 open question) — official GW2 API `/v2/skills` (no `type` field) + manual mapping OR community dataset (gw2efficiency / discretize) | S (decision) |
| B.2 | Build the `gw2_skills` library (`libs/gw2_skills/`) with `SkillMetadata` Pydantic class (`id + name + type + icon_url + categorisation: boon/utility/elite/heal/CC`) | M |
| B.3 | Bootstrap catalog from chosen source (one-time script + versioned JSON dataset under `libs/gw2_skills/src/gw2_skills/data/`) | M |
| B.4 | Cargo-cult the skills DB into `apps/api` startup (load + cache via `functools.lru_cache(maxsize=1)` keyed on the catalog hash) | S |
| B.5 | Expose `gw2_skills.lookup_skill(id)` to libs + apps with a `SkillNotFoundError` for unknown IDs (defensive) | S |
| B.6 | Ship v0.10.20+ as a separate library + add to `pyproject.toml` workspace + verify cross-library typechecks | S |
| B.7 | Add 5+ hermetic tests for the catalog fixture (abi / canonical 500 skills subset) | S |

**Total:** M-L effort (7 sub-blocks, ~600 LoC across new library + bootstrap script + cache wiring + tests).

### Blocker C — Role classifier (per the §3.1 heuristics)

**Location:** §3.1 lists 7 role thresholds (HEAL / DPS / STRIP / SUPPORT / CC / TANK / BOONS). The classifier v1 lite is SHIPPED pre-F17 cycle via `libs/gw2_analytics/src/gw2_analytics/role_detection.py`; the v2 bump (refined thresholds + offline calibration) is a maintenance-cycle deliverable. F17 design §9 step 3's downstream caller follows the same shape.

**Why it's "blocked":** the role classifier is not blocked on external dependencies — it's pure analytic logic over the per-player roll-ups. The v1 lite shipped during the Wave 6/7 cycle, ahead of full stable Wave 3 R.1-R.4 per-player roll-ups, by relying on the pre-Wave 3 roll-up schema (a documented subset). The v2 bump is gated on stable R.1-R.4 inputs (Blockers A + status) and offline corpus calibration.

**Sub-blocks:**

| Sub-block | Description | Effort |
|---|---|---|
| C.1 | **v1 lite DELIVERED** via `role_detection.py` (full 7-role set, pure function, deterministic, no I/O). v2 refinement pending R.1-R.4 stability. | M |
| C.2 | v2 thresholds need calibration against a representative historical zevtc corpus (N≥20 boss fights across M=4 archetypes: raid / fractal / strike / wvw) once R.1-R.4 stable. | L (calendar time, not code) |
| C.3 | **Shipped** via `libs/gw2_analytics/tests/test_role_detection.py` (5+ hermetic tests, in-progress additions as v2 lands). | S |

**Total:** M-cum-L effort (3 sub-blocks; ~150 LoC v1 lite + ~50 LoC v2 polish + offline v2 calibration).

- **v0.10.21** (or whenever): ship Blocker A (XL) + Blocker B part 1 (M-L). ~1800 LoC, backend-only.

## 5. Delivery order (proposed)

**If authorised, ship in this order:**

1. **Phase 9 step 4 — statechange parser extension** (Bl A). The statechange blocker is the largest single LoC delta + the highest dep-blocking nature (R.1-R.4 cannot ship without it).
2. **Skills DB bootstrap** (Bl B). Ships as a standalone library (low outward impact); consumed by R.1-R.4 once available.
3. **Roller aggregators + role classifier** (R.1-R.4 + Bl C). Backend-only; no frontend coupling.
4. **Routes + schemas** (Rt.1-Rt.4). Backend API surface; the frontend consumes via TanStack Query once shipped.
5. **AG Grid frontend tables** (W.2-W.12). Frontend-only with backend integration verification.
6. **Visual regression baseline + Playwright e2e** (W.11-W.12). Closes the cycle.

## 6. Cut list (if v0.10.21 budget is exceeded)

If the cycle team is asked to F17 as a single v0.10.21 cycle and the mimo-half budget can't cover XL+, the natural cuts are:

- **Cut W.5 (Boons table)** — the Boons table is the most-complex AG Grid component (dynamic "other boons" column + 6 fixed boon columns + the 2 horizontal bars). Ship the Damage / Heal / Defense tables in v0.10.21; ship Boons in v0.10.22.
- **Cut X.3 (cycle-end audit)** — defer to v0.10.22 (annotate the v0.10.21 CHANGELOG entry as "F17 partial — 3 of 4 tables; Boons + cross-doc audit deferred to v0.10.22").
- **Cut W.11 (Playwright e2e)** — defer to v0.10.22 (ship vitest for the 3 tables; Playwright follows the v0.10.18 D3 precedent of e2e-after-vitest).

If ALL cuts are made, v0.10.21 ships in M-L effort (Bl A + Bl B + R.1-R.4 + Rt.1-Rt.4 + W.1-W.4 + W.6-W.10 + W.12 = ~3500 LoC vs the original ~6400 LoC).

## 7. Open questions (locked + pre-flagged for the implementation cycle)

Per the §11 design-doc open questions:

1. **Skills DB source** — official GW2 API + manual mapping, or community dataset? Decision is part of Bl B.1.
2. **Role vocabulary** — the exact set of `roles: list[str]` values. Final set comes from Bl C.2 calibration.
3. **"Autres boons" rendering** — dynamic columns, collapsed tooltip, or top-3 column? Locked default: dynamic columns (decision deferred to W.5 implementation).
4. **"Time on the ground" semantics** — locked default: time downed (downed state duration).
5. **Page route** — locked default: integrate into `/fights/[id]` (the §12 Notes recommendation; not a separate page).

## 8. Notes for the implementation cycle

- The `Event` discriminated union (in `libs/gw2_core/src/gw2_core/models.py`) will grow from the current 3 members + 2 spike-stub members (`BoonApplyEvent`, `CCEvent`) to ~10-12 final members. The `Field(discriminator=\"event_type\")` contract is forward-compat: the JSONL reader will dispatch on the new types without code changes.
- The per-player roll-ups re-use the existing `_compute_contributions` helper from v0.7.0 — the helper is already source-side attribution so the new tables can be added without a refactor of that helper.
- The "kills" / "down" / "boons" / "CC" counts are pre-aggregated counts (per player per fight), not rates. The frontend can derive rates if needed (typically the analyst wants both: a count + a count/s).
- The role classifier is the **only heuristic** in the cycle; everything else is deterministic aggregation. Calibrate it last (after the per-player roll-ups are stable) so the role set has a stable input.
- Defense table cannot ship without statechange parser (downed / death / dodge / block / interrupt statechange events). The §6 cut list above DOES NOT cut the Defense table from the minimum viable v0.10.21 — it cuts Boons instead, which has more replaceable fallback (a single "boon count" cell per player) vs Defense (no fallback).

## 9. Forward cadence post-v0.10.19

If the maintener authorises F17:

- **v0.10.21**: Bl A (statechange) + Bl B (skills DB) + R.1-R.4 (rollers) + Rt.1-Rt.4 (routes). **Backend-only cycle.** ~3550 LoC. Effort: L.
- **v0.10.22**: W.1-W.12 (frontend) + cycle-end audit + visual baseline + Playwright e2e. **Frontend-only cycle with backend consume-validation.** ~2700 LoC. Effort: XL+.
- **v0.10.23**: F17 long-tail (calibration of role thresholds against the user's logs + minor cosmetic tweaks). **Maintenance cycle.** ~150 LoC. Effort: S-M.

If the maintener defers F17 to v0.10.23+:

- **v0.10.20**: M5 (F17 long-tail) + M6 (AG Grid bundle fix) + M7 (CHANGELOG bucketing). The §1.2 list rotates; F17 stays as item 1.
- **v0.10.21+**: F17 ships when authorised.

## 10. Summary

F17 is achievable on the declared XL+ effort budget IF the cycle team:

1. **Closes M8 first** (v0.10.19 mimo-half PRIMARY).
2. **Ships Bl A + Bl B as 1 cycle** (the statechange parser + skills DB waterfall is the natural backend-only unit of work; saves cross-cycle dep handoff overhead).
3. **Ships R + Rt + W over 2 more cycles** (the rollout pattern).
4. **Uses the §6 cuts** if the mimo-half budget can't cover the declared scope.

The spike doc AUTHORITATIVELY spec's the order + estimate + cut list. The design doc remains the source of truth for the table shapes + column contracts. Future cycle docs (v0.10.21+ release notes + audits) cite this spike doc as the impl-order basis.
