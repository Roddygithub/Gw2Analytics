# v0.9.0 combat readout design — "Lecture combat"

**Status:** Draft (post-v0.8.0)
**Backend dependency:** parser extension (statechange), skills DB, role classifier
**Web dependency:** new AG Grid components + 4 table layouts
**Target:** v0.9.0 (backend first, then web as a follow-up minor — the v0.7.0 / v0.7.1 / v0.8.0 pattern)

---

## 1. Motivation

The v0.7.0 / v0.7.1 / v0.8.0 web layers ship a player-centric
surface but **no structured combat readout** — the analyst sees
per-target / per-skill roll-ups but cannot read a fight the way
Elite Insights or arcdps' personal stats panel can: a 4-table
summary that shows at a glance what each player did, organised by
aspect (Damage / Heal / Boons / Defense).

The v0.9.0 cycle adds this readout as **4 AG Grid tables sharing
5 first-columns**, each table a self-contained roll-up. The user
spec is below.

## 2. Shared columns (all 4 tables)

| # | Column | Source | Notes |
|---|---|---|---|
| 1 | **Sub-groupe** | `OrmAgent.subgroup` | integer → label mapping (e.g. `1` → "Sub 1", `2` → "Sub 2"). |
| 2 | **Nom du personnage** | `OrmAgent.name` | |
| 3 | **Spécialisation jouée** | `PlayerProfile.elite_spec` (or `profession` if no elite) | Rendered with the Tango Medium icon (see §8). |
| 4 | **Icône commandant** | `OrmAgent.is_commander` | Crown icon, only when the flag is set. |
| 5 | **Rôle(s) du joueur** | `roles: list[str]` from the role classifier | Multi-role set (e.g. `["DPS", "STRIP"]`). See §3.1. |

The 5 shared columns live in a `<PlayerReadoutBase>` wrapper
component so the 4 tables inherit them without duplication.

## 3. Table 1 — Damage

Per-player damage contribution roll-up.

| Column | Sub-element | Source |
|---|---|---|
| **DPS total** | | `sum(total_damage) / duration_s` |
| | DPS power | per-skill power/condi split from skills DB (see §5) |
| | DPS condi | same |
| **Strips** | | sum of `BuffRemovalEvent.buff_removal` where `source_agent_id = player` |
| **CC appliqués** | | statechange count (stun / daze / knockback / pull / launch / float / sink) where `source = player` |
| **Down contribution DPS** | | `sum(damage)` where target is in `downed` state at event time |
| **Kills** | | statechange count where `source = player` and target goes from `alive` to `dead` |

### 3.1 Role classifier (backend)

The "rôles" column is a `set[str]` derived from the per-player
roll-ups. The heuristic is calibrated against the user's local
"thousands of logs" — see §5 for the open question on the
vocabulary.

Indicative thresholds (calibration TBD):
- `HEAL` — `barrier/s + heal/s` > 30% of the squad's total
- `DPS` — `dps` > 50% of the squad's average
- `STRIP` — `total_buff_removal` > 30% of the squad's total
- `SUPPORT` — `boon_out / duration_s` > 1.0 (boons applied per second)
- `CC` — `cc_count` > 5
- `TANK` — `damage_taken` > 2× squad's average AND `block_count` > 20
- `BOONS` — same as `SUPPORT` (synonym in some communities)

## 7. Clarifications (locked from the brainstorming sessions)

- **« Spés »** = the spec being played (core profession OR elite
  spec), treated as a single field. Not core-only, not elite-only.
- **« Rôles »** = a **SET** (`roles: list[str]`), e.g.
  `["DPS", "STRIP"]`. A single player can be DPS + STRIP + Support
  simultaneously. The set is backend-computed, never client-typed.
- **« Down contribution DPS »** = damage dealt to enemies
  **already in the downed state** (not damage the player deals
  while downed). Requires the parser to track the `downed` state
  per target across events.
- **« Cleave »** = the **number of kills the player contributes
  to** (NOT the standard GW2 AoE multi-target hits). Computed as
  statechange events where `source = player` and the target
  transitions from `alive` to `dead`.
- **« Cleanses » / « Breakstunt »** = statechange counts. The
  Firebrand F5 "Epilogue: Ashes of the Just" is the canonical
  Breakstunt example.
- **« Heal + Barrier »** = barrier **must be separable** from
  heal. The `HealingEvent` model gains a `barrier: int` field
  distinct from `healing: int` (currently they are conflated).
  This is the smallest v0.9.0 change to the existing model.
- **« Spés » icons** = Tango Medium from the official wiki (see
  §8). The 9 core professions + ~30 elite specs = ~40 icons.

## 8. Icon source

Style: **Tango Medium** from
[`Guild_Wars_2_Wiki:Profession_icons`](https://wiki.guildwars2.com/wiki/Guild_Wars_2_Wiki:Profession_icons).

Two asset sets:
- 9 core profession icons → `web/public/icons/professions/`
- ~30 elite spec icons → `web/public/icons/specializations/`

Naming convention (inferred from the wiki):
- `Profession_<name>_tango.png`
- `Specialization_<name>_tango.png`

**Licence risk:** ArenaNet's wiki Terms of Use. Acceptable for a
personal analytics project; the practice is standard in the GW2
community (gw2efficiency, discretize, snowcrows do the same). If
this project ever becomes public, the asset strategy may need
to change (link to `Special:FilePath` at render time instead of
bundling).

**Acquisition path:** 50-line Node spike that downloads the icons
once into `web/public/icons/`, then committed to the repo. Re-run
only if ArenaNet updates the icon set.

## 9. Build sequence (dependency-ordered)

1. **Statechange parser** (`libs/gw2_evtc_parser`)
   - Prerequisite for ~80% of the tables.
   - Parse the ~150 statechange IDs from arcdps' documented set
     (reference: Elite Insights' C# `StateChange` enum).
   - Add 9 new `Event` subclasses in `libs/gw2_core`:
     `BoonApplyEvent`, `ConditionRemoveEvent`, `CCEvent`,
     `DownEvent`, `DeathEvent`, `DodgeEvent`, `BlockEvent`,
     `InterruptEvent`, `StunBreakEvent`.
   - The JSONL storage path already uses
     `TypeAdapter(Event).validate_json(line)` so the dispatch is
     forward-compat.

2. **Skills DB** (`libs/gw2_skills` — new library)
   - Static JSON dataset of `id → {name, type: power | condi | hybrid, ...}`.
   - ~1000 entries, versioned alongside the GW2 API.
   - Source: official GW2 API `/v2/skills` (no type field) +
     manual mapping, OR a community dataset (gw2efficiency,
     discretize). Decision deferred to the spike.

3. **Role classifier** (`libs/gw2_analytics`)
   - New module `PlayerRoleClassifier.classify(roll_ups) -> set[str]`.
   - Heuristic over the per-player roll-ups (§3.1).
   - Calibrate thresholds against the user's local "thousands of
     logs" before the release.

4. **Per-player roll-ups** (`libs/gw2_analytics`)
   - 4 new aggregators: `PlayerDamageAggregator`,
     `PlayerHealAggregator`, `PlayerBoonsAggregator`,
     `PlayerDefenseAggregator`.
   - Each outputs a `PlayerXxxRow` (the per-table column set).
   - All aggregators re-use the existing `_compute_contributions`
     helper from v0.7.0 (source-side attribution already correct).

5. **Backend routes** (`apps/api`)
   - Two options:
     - (a) 4 sibling routes:
       `GET /api/v1/fights/{id}/players/damage`,
       `.../heal`, `.../boons`, `.../defense`.
     - (b) 1 unified route:
       `GET /api/v1/fights/{id}/readout` returning all 4 tables.
   - Default to (b) for fewer round-trips; the page renders all
     4 sections from a single fetch (mirrors the v0.7.1
     `Promise.allSettled` pattern for fights/[id]).

6. **Frontend tables** (`web`)
   - 4 new AG Grid components:
     `<PlayerReadoutDamage>`,
     `<PlayerReadoutHeal>`,
     `<PlayerReadoutBoons>`,
     `<PlayerReadoutDefense>`.
   - All share the 5 first-columns from `<PlayerReadoutBase>`.
   - Page route: new `/fights/[id]/readout` page OR integrate
     into the existing `/fights/[id]` (the latter avoids
     splitting the analyst's drill-down flow).

---

## 10. Out of scope (v0.9.0)

- **Distance moyenne au commandant** — V1.3 EVTC does NOT include
  per-event positions. The agent table carries only the
  initial position at fight start. There is no per-event
  coordinate in the binary format. Workarounds (range-based
  inference from skill casts, OCR of screenshots, parsing the
  separate arcdps `.log` file) are too approximate to be
  useful. Drop for v0.9.0; revisit when ArenaNet publishes an
  EVTC V2 with per-event positions.

- **Multi-account comparison** — separate v0.9.x feature (already
  on the §1 ROADMAP list).

- **Real-time DPS meter** — separate v0.9.x feature (already on
  the §1 ROADMAP list).

## 11. Open questions (for the implementation cycle)

1. **Skills DB source** — official GW2 API + manual mapping, or
   community dataset? Each has a different maintenance profile.
2. **Role vocabulary** — the exact set of `roles: list[str]`
   values. The §3.1 list is indicative; the final set comes from
   the calibration against the user's logs.
3. **"Autres boons" rendering** — the 6 named boons get fixed
   columns; the other ~34 boons could be (a) dynamically
   expanded columns (AG Grid supports this), (b) collapsed into
   a single "Other boons (N)" cell with a tooltip listing the
   breakdown, or (c) a 7th column "Top 3 other boons" that
   shows the 3 most-applied.
4. **"Time on the ground" semantics** — "time downed" (downed
   state duration) is the natural reading. Confirm.
5. **Page route** — new `/fights/[id]/readout` page, or
   integrate into the existing `/fights/[id]`? Default to
   integrate (no flow split).

## 12. Notes for the implementation cycle

- The `Event` discriminated union (in
  `libs/gw2_core/src/gw2_core/models.py`) will grow from 3
  members to ~12. The `Field(discriminator="event_type")`
  contract is forward-compat: the JSONL reader will dispatch
  on the new types without code changes.
- The per-player roll-ups re-use the same `_compute_contributions`
  helper from v0.7.0 — the helper is already source-side
  attribution so the new tables can be added without a
  refactor of that helper.
- The "kills" / "down" / "boons" / "CC" counts are
  pre-aggregated counts (per player per fight), not rates.
  The frontend can derive rates if needed (typically the
  analyst wants both: a count + a count/s).
- The v0.9.0 cycle ships the **backend first**, then the
  web layer as a follow-up minor — the
  v0.7.0 / v0.7.1 / v0.8.0 pattern (semver convention:
  additive changes only, no breaking contract changes).
- The role classifier is the **only heuristic** in the
  cycle; everything else is deterministic aggregation.
  Calibrate it last (after the per-player roll-ups are
  stable) so the role set has a stable input.

## 13. Default sort (locked)

**Primary sort key (all 4 tables):** `subgroup ASC` — integer 1, 2,
3, ... (ascending). Group by squad first.

**Tiebreaker (per table):**

| Table | Tiebreaker | Rationale |
|---|---|---|
| Table 1 — Damage | `dps DESC` | Top damage dealer of each squad surfaces first. |
| Table 2 — Heal | `hps DESC` (= `heal/s + barrier/s`) | Top healer of each squad surfaces first. |
| Table 3 — Boons | `boon_out_rate DESC` (= `boons_applied / duration_s`) | Top boon provider of each squad surfaces first. |
| Table 4 — Defense | `damage_taken DESC` | Most-targeted player of each squad surfaces first (defensive load is the leading indicator). |

**Re-sort behaviour:** the analyst can re-sort by clicking any
column header (AG Grid Community default). The « default sort »
above is the state on first page load only. Switching to a pure
DPS-pure ranking is one click away on the `dps` column.

**Why subgroup-first (not role-first or pure-dps):** the WvW raid
leader's mental model is **squad-centric** — each subgroup is a
distinct unit (« the bomb », « the main zerg », « the home
defenders »). Grouping by squad first lets the analyst read the
table top-to-bottom as: « squad 1, top contributor first; squad
2, top contributor first; ... ». The role-based grouping was
considered and rejected: WvW roles are softer than PvE (a single
player is often `DPS + STRIP` simultaneously) and the squad unit
is the primary organisation in the field.
