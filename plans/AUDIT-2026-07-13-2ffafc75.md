**Distribution by sub-bucket**:
- **K1 (5): Arq-Worker connectivity** — `test_uploads_arq.py` fixtures assume `ALLOW_INREQUEST_PARSE_FALLBACK=1` + isolated Arq pool, but the live docker-compose stack spawns a real Redis-backed Arq worker that becomes reachable from inside the test process. The pytest fixture's monkeypatching leaks at `conftest.py::_disable_arq_for_tests` global.
- **K2 (4): IP-routing/SSRF gate semantics** — `test_webhooks_e2e.py` SSRF block tests (private/literal/link-local/IPv6) assert HTTPS requests to literal private addresses get a 4xx. On the live docker-compose network namespace, the webhooks SSRF validator's socket-level checks behave differently from the v0.9.2 webhook SSRF module's monkeypatched `socket.getaddrinfo` baseline.
- **K3 (2): DNS-resolver-pool saturation / latency budget** — `test_webhooks_dns_under_attack.py::test_pool_saturation_gracefully_returns_422` + `test_webhooks_getaddrinfo_timeout.py::test_getaddrinfo_timeout_returns_422` require bounded `ThreadPoolExecutor` + timeout under load. The current `_DNS_EXECUTOR = ThreadPoolExecutor(max_workers=1)` is sized correctly but stress tests assert ~50ms latency; on a CI box with 100% CPU contention, the test sees the request hang instead of 422.

**Priority for v0.10.19 mimo-half budget: PRIMARY.** Closes test-substrate health before F17 Combat readout can ship reliably. The M8 ship precedes everything else in §1.2.

### Guard rail — NOT a regression

> None of the 11 failures indicate a regression in core application logic; they are purely isolation leaks and test-env mismatches where the test suite's fake DNS, literal IP assumptions, and Redis mocks collide with the host's live docker-compose substrate.

The production code paths (POST /api/v1/webhooks SSRF gate, GET /api/v1/fights/{id}/events Arq fallback, etc.) work correctly in live deployment. v0.10.19's M8 fix-up will land test-substrate corrections (conftest isolation + DNS monkeypatch restoration + Arq mock-pool parity) without touching production logic.
