## Context

### Real-payload testing findings (2026-07-09)

The user supplied **1,605 WvW `.zevtc` log files (2.6 GB)** organized by 11 character/account names. File size distribution:

| Plage | Count | % |
| --- | --- | --- |
| >1MB | 784 | 48% |
| >5MB | 104 | 6% |
| >20MB | 6 | 0.4% |
| **Total** | **1,605** | **100%** |

**Test sequence + observed state:**

1. Smoke test: 1 sequential upload (`20251205-233553.zevtc`, 164K) → ✅ fight ingested, queryable, `process_parse` completed
2. Bulk test: 8 parallel uploads (4 accounts × 2 files, ThreadPoolExecutor max_workers=4) → ❌ all 8 stuck on `pending` for 20s, no fight_ids
3. `GET /api/v1/fights?limit=5` → only 1 fight exists in DB, with 0 player agents and 0 duration
4. `tail -n 200 /tmp/fastapi.log` → 253K chars dominated by `psycopg.errors.UndefinedColumn: column webhook_subscriptions.secret does not exist` (one per scheduler tick, every ~5s)
5. `psql \d webhook_subscriptions` → table has `ciphertext` column, NO `secret` column
6. `alembic current` → head = `0009_webhook_secret_at_rest`

**Diagnosis (validated by Gemini 2.5 Pro):**

- **Bug #1**: SQLAlchemy holds the pre-migration ORM mappings in the running Python process. The DB is correct; the in-memory class registry is stale. A lifespan startup check that compares `alembic_version.version_num` to `alembic.script.ScriptDirectory.get_current_head()` would crash Uvicorn at boot with an actionable error message instead of letting the operator discover the drift via log spam minutes/hours later.
- **Bug #2**: `BackgroundTasks` runs in a single-process thread pool under the GIL. 8 CPU-bound parser tasks serialize on the GIL → none finish in 20s. Moving to an Arq worker process (separate GIL, separate core pool) is the canonical fix; the existing webhook scheduler pattern (sync function in `lifespan_scheduler`) is the right template.

**Side-finding (not in this plan):** `dispatch_for_upload` is scheduled concurrently with `process_parse` in the same `BackgroundTasks` (see `routes/uploads.py` line 92-105), but `dispatch_for_upload` short-circuits if `upload.status != COMPLETED` — a race where the dispatch returns 0 deliveries because the parse hasn't finished yet. Step 3 of this plan chains `dispatch_for_upload` INSIDE the Arq job (right after `process_parse` returns) to close this race at the same time.

### Step 2: Schema-drift guard at startup (S)

- New `apps/api/src/gw2analytics_api/schema_guard.py`:
  ```python
  from __future__ import annotations
  import logging
  from alembic.config import Config as AlembicConfig
  from alembic.script import ScriptDirectory
  from sqlalchemy import text
  from gw2analytics_api.database import get_sessionmaker

  logger = logging.getLogger(__name__)

  def check_schema_drift() -> None:
      """Compare alembic head to live DB alembic_version. Raise on drift.

      A drift means the Uvicorn process was started before the latest
      migration ran (the in-memory SQLAlchemy ORM registry is stale,
      queries reference columns that no longer exist, the scheduler
      spams UndefinedColumn errors). Crashing at startup surfaces the
      problem in the operator's terminal instead of 5s later in
      /tmp/fastapi.log.
      """
      cfg = AlembicConfig("alembic.ini")
      head = ScriptDirectory.from_config(cfg).get_current_head()
      with get_sessionmaker()() as db:
          actual = db.execute(text("SELECT version_num FROM alembic_version")).scalar_one_or_none()
      if actual != head:
          raise RuntimeError(
              f"Schema drift detected: database is at {actual!r}, "
              f"code is at {head!r}. Did you forget to restart the API "
              f"after running migrations? (Set SKIP_SCHEMA_GUARD=1 to "
              f"bypass in emergencies.)"
          )
      logger.info("schema drift check: ok (head=%s)", head)
  ```
- `apps/api/src/gw2analytics_api/main.py`: call `check_schema_drift()` at the top of `lifespan`, BEFORE `lifespan_scheduler` starts:
  ```python
  @asynccontextmanager
  async def lifespan(_app):
      import os
      if not os.environ.get("SKIP_SCHEMA_GUARD"):
          check_schema_drift()
      scheduler_task = asyncio.create_task(lifespan_scheduler(_open_session))
      try:
          yield
      finally:
          scheduler_task.cancel()
          with suppress(asyncio.CancelledError):
              await scheduler_task
  ```
- **Verify** (post-Step-2):
  - `cd apps/api && PYTHONPATH=. SECRETS_KEK='...' uv run python -c 'from gw2analytics_api.main import app'` → starts cleanly (drift = 0)
  - Simulate drift: `psql -c "UPDATE alembic_version SET version_num='0008_payload_bytes'"` → restart API → expect `RuntimeError: Schema drift detected` in the first 1-2 lines of uvicorn output
  - `unset` the simulated drift + restart → clean startup

### Step 3: Arq parser worker (M)

- New `apps/api/src/gw2analytics_api/workers/parser_worker.py`:
  ```python
  from __future__ import annotations
  import logging
  from uuid import UUID
  from arq.worker import Retry
  from gw2analytics_api.database import get_sessionmaker
  from gw2analytics_api.services import process_parse
  from gw2analytics_api.workers.webhook_dispatch import dispatch_for_upload

  logger = logging.getLogger(__name__)

  async def parse_job(ctx, upload_id: str, raw_bytes: bytes) -> None:
      """Arq job: parse one upload + dispatch webhooks (chained).

      Replaces the dual ``BackgroundTasks.add_task(process_parse, ...)``
      + ``BackgroundTasks.add_task(dispatch_for_upload, ...)`` in
      routes/uploads.py. Chaining inside one job closes the race
      where dispatch short-circuits if the parse hasn't flipped the
      status to COMPLETED yet.

      Runs the sync parse + dispatch in ``asyncio.to_thread`` so the
      event loop stays responsive for the next job pickup.
      """
      sf = get_sessionmaker()
      try:
          await asyncio.to_thread(process_parse, sf, UUID(upload_id), raw_bytes)
      except Exception:
          logger.exception("parse_job parse failed for upload %s; not dispatching webhooks", upload_id)
          raise Retry(defer=ctx["job_try"] * 5)  # arq exponential backoff
      try:
          await asyncio.to_thread(dispatch_for_upload, sf, UUID(upload_id))
      except Exception:
          logger.exception("parse_job dispatch failed for upload %s (parse already committed)", upload_id)
          # do NOT retry — the parse committed, the user already sees
          # the fight in /fights; missing webhooks are an operational
          # concern (the operator can re-dispatch via a manual CLI)
  ```
- New `apps/api/src/gw2analytics_api/workers/parser_settings.py`:
  ```python
  from __future__ import annotations
  from arq.connections import RedisSettings
  from gw2analytics_api.workers.parser_worker import parse_job

  class WorkerSettings:
      functions = [parse_job]
      redis_settings = RedisSettings(host="localhost", port=6379)
      max_jobs = 2  # CPU-bound parser; pin to keep memory bounded
      job_timeout = 600  # 10 min ceiling for a 20MB log
  ```
- Add `__main__` to `parser_settings.py` so `arq gw2analytics_api.workers.parser_settings.WorkerSettings` spins up the worker (arq's standard CLI convention).
- **Verify** (post-Step-3):
  - `cd apps/api && PYTHONPATH=. SECRETS_KEK='...' arq gw2analytics_api.workers.parser_settings.WorkerSettings` → worker starts, logs "Found job functions: ['parse_job']"
  - `redis-cli ping` → `PONG`

### Step 4: Wire Arq pool to upload route (S)

- `apps/api/src/gw2analytics_api/main.py` lifespan: add Arq pool init/teardown:
  ```python
  from arq import create_pool
  from gw2analytics_api.workers.parser_settings import WorkerSettings

  @asynccontextmanager
  async def lifespan(_app):
      if not os.environ.get("SKIP_SCHEMA_GUARD"):
          check_schema_drift()
      _app.state.arq_pool = await create_pool(WorkerSettings.redis_settings)
      scheduler_task = asyncio.create_task(lifespan_scheduler(_open_session))
      try:
          yield
      finally:
          scheduler_task.cancel()
          with suppress(asyncio.CancelledError):
              await scheduler_task
        await _app.state.arq_pool.aclose()
  ```
- `apps/api/src/gw2analytics_api/routes/uploads.py`:
  - Add `from fastapi import Request`
  - Change `def create_upload(background_tasks: BackgroundTasks, ...)` to `def create_upload(request: Request, ...)`
  - Replace the two `background_tasks.add_task(...)` calls with:

    await request.app.state.arq_pool.enqueue_job("parse_job", str(upload.id), raw)

  - For the idempotency re-parse path (`existing.status == "failed"`), enqueue the same job (idempotent: `parse_job` checks the upload row first).

## Maintenance note

- The `SKIP_SCHEMA_GUARD=1` escape hatch is intentionally NOT a permanent flag — it's for `docker compose run` migrations that share a network with a live Uvicorn (rare). Add a comment in `main.py` warning that disabling the guard in production is a temporary measure.
- The Arq worker is a NEW process; the existing `lifespan_scheduler` (webhook retry/DLQ) stays in-process for now. Migrating `webhook_scheduler` to Arq is a v0.11 unification candidate (the scheduler is a 1s-tick loop, would map to a recurring Arq cron job, but it's out of scope for v0.10.1).
- `max_jobs=2` is conservative; raise to `os.cpu_count()` if the 20MB+ log files in the user's archive ever become the norm (currently 6 / 1,605 = 0.4% of files).

## Escape hatches

- **STOP** if `arq>=0.25` introduces a breaking API change in `WorkerSettings.functions` (it's been stable since 0.20; verify with `uv add arq` first). Fallback: pin `arq==0.25` in pyproject.
- **STOP** if `test_uploads_e2e.py` fails with `RuntimeError: Schema drift detected` AFTER the migration is applied. The most likely cause: the test DB and the production DB have different `alembic_version` rows. Verify with `cd apps/api && uv run alembic current` and `psql -c 'SELECT * FROM alembic_version'`.
- **STOP** if `arq` worker OOMs on a 39MB log file. Bump the worker's `--max-jobs 1` + add `job_timeout=900` (15min). The 20MB+ files are 0.4% of the archive; a 15min ceiling is well above the observed parse time for a 5MB log (~30s).
- **STOP** if the chained `dispatch_for_upload` inside the Arq job causes idempotency issues (e.g. a retry of `parse_job` dispatches the SAME webhook twice). The `dispatch_for_upload` checks `upload.status == COMPLETED` and skips if not, but a re-dispatch after a partial commit could fire twice. Mitigation: the Arq `Retry` is only raised on `process_parse` failure (not on `dispatch_for_upload` failure), so the chain is one-shot. If double-dispatch is observed in production, add a `dispatched_at` column to `OrmUpload` and a `WHERE dispatched_at IS NULL` guard.
- **ROLLBACK** (if v0.10.1 ships and breaks production): revert the commit, restart Uvicorn (the in-flight process is unaffected), and the existing `BackgroundTasks` path resumes. No data loss (Arq jobs that completed wrote to the same DB + MinIO; in-flight jobs are dropped on Arq shutdown and the operator can re-POST the file — SHA-256 idempotency makes the re-POST safe).
