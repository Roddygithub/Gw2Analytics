# Plan 105 (v0.9.34) ŌĆö `alembic/env.py` `compare_type=True` + `compare_server_default=True` for both online + offline modes

## Files touched
- `apps/api/alembic/env.py` (3-line addition to `run_migrations_offline::context.configure()` + same for `run_migrations_online::context.configure()`)
- `apps/api/tests/alembic/test_env_configure.py` (NEW ŌĆö 4 hermetic tests verifying the configure flags are set)

## Findings (audit)

- `apps/api/alembic/env.py::run_migrations_offline::context.configure(url=..., target_metadata=..., literal_binds=True)` ŌĆö missing `compare_type=True` + `compare_server_default=True`.
- `apps/api/alembic/env.py::run_migrations_online::context.configure(connection=..., target_metadata=...)` ŌĆö missing the same 2 flags.
- Effect: `alembic revision --autogenerate -m "..."` produces an EMPTY migration when the diff between ORM metadata and live-DB schema is a column-type or server-default change (e.g., `String(length=64)` ŌåÆ `String(length=128)` like 0002, or adding `server_default=sa.text("now()")` like 0006). The alembic autogen comparison is type-blind by default; it only sees "the column is named X with the same nullable flag" and reports zero diff.
- The drift hazard: a future ORM-side change to a column type or default would silently NOT generate the corresponding alembic migration; the live DB would diverge from the ORM metadata; production behaviour diverges from test behaviour; the divergence surfaces only at runtime (a `sa.Uuid()` vs missing `Uuid` would either 422 at insert or silently fail to validate).
- Plan 061 v0.9.19 flagged this exact problem in the audit log (the file-content scan at that time identified the env.py configure calls as missing the flags). The plan called for adding the flags; the change was NOT made.
- The 2 affected migrations in the workspace that WOULD have been autogenerated correctly with these flags:
  - `0002_agent_identity_columns.py::alter_column("name", existing_type=String(length=64), type_=String(length=128))` ŌĆö a type-length change; `--autogenerate` would have detected it.
  - `0006_webhooks.py::Column(... server_default=sa.text("now()"))` ├Ś 3 ŌĆö a server-default addition; `--autogenerate` would have detected each.

## Fix

1. `apps/api/alembic/env.py::run_migrations_offline` ŌĆö replace:

   ```python
   def run_migrations_offline() -> None:
       url = config.get_main_option("sqlalchemy.url")
       context.configure(url=url, target_metadata=target_metadata, literal_binds=True)
       with context.begin_transaction():
           context.run_migrations()
   ```

   with:

   ```python
   def run_migrations_offline() -> None:
       url = config.get_main_option("sqlalchemy.url")
       context.configure(
           url=url,
           target_metadata=target_metadata,
           literal_binds=True,
           # Re-enable alembic's type + server-default diff detection.
           # Without these flags ``alembic revision --autogenerate``
           # produces an EMPTY migration when the diff is a column
           # type change or a server-default change -- the autogen
           # comparison is type-blind by default and only sees
           # "the column exists with the same nullable flag".
           #
           # ``compare_type=True`` catches changes like
           # ``String(length=64) -> String(length=128)``.
           # ``compare_server_default=True`` catches additions like
           # ``server_default=sa.text("now()")``.
           #
           # Both flags combine with the existing ``target_metadata``
           # to drive the autogen output. False positives are
           # suppressed by the comment block on the flagged column
           # that documents why a specific type / default choice was
           # made (the v0.8.4 migration 0005 docstring is the
           # canonical pattern).
           compare_type=True,
           compare_server_default=True,
       )
       with context.begin_transaction():
           context.run_migrations()
   ```

2. Same arg block on `run_migrations_online::context.configure()`:

   ```python
   def run_migrations_online() -> None:
       connectable = engine_from_config(
           config.get_section(config.config_ini_section, {}),
           prefix="sqlalchemy.",
           poolclass=pool.NullPool,
       )
       with connectable.connect() as connection:
           context.configure(
               connection=connection,
               target_metadata=target_metadata,
               compare_type=True,
               compare_server_default=True,
           )
           with context.begin_transaction():
               context.run_migrations()
   ```

3. NO `alembic.ini` change needed ŌĆö the flags are passed directly to `context.configure()` per alembic's API. The `alembic.ini` `sqlalchemy.compare_type` global is an alternative, but the `context.configure()` kwargs are the canonical pattern.

## Rejected alternatives

- **Set the flags globally in `alembic.ini`** ŌĆö works but is less discoverable than the in-file kwargs. A future contributor looking at `env.py` would miss the global setting. The kwargs pattern is the alembic-recommended approach. REJECTED.
- **Add `compare_type=True` only (skip `compare_server_default=True`)** ŌĆö leaves the second drift hazard in place (server-default changes still go un-detected). Both are needed. REJECTED.
- **Add a runtime warning when `compare_type=False`** ŌĆö diagnostic only; doesn't fix the underlying problem. The kwargs are the fix. REJECTED.
- **Skip the offline-mode update** ŌĆö `--autogenerate --sql` (the SQL-mode autogen) uses the offline path. Skipping it leaves the SQL-mode autogen blind. Both paths need the flags. REJECTED.
- **Move the flags to the `Base.metadata` config (e.g. via the declarative base)** ŌĆö alembic doesn't expose metadata-level autogen flags; the kwargs to `context.configure()` are the only canonical entry point. REJECTED.
