## Why v0.10.16 didn't ship (the 5 actual causes)

1. **Maintainer's "no MiMo credits for today" decision.** The miMo working agent had no credits remaining in today's session budget; per the project's split-cycle convention (the prior v0.10.14 cycle was a miMo-half cycle), MiMo execution was the delivery surface for D1-D4. Without MiMo credits, the briefs land on disk but no D1-D4 commits ship.
2. **No parent-agent (Buffy) full-cycle picking-up.** Per the project's split-cycle convention, the parent agent scopes + delegates + handles close-out but does NOT execute implementation work. v0.10.16's deferred scope was miMo-only; the parent had no implementation surface to take over.
3. **No fallback surface for tests-only cycle.** A pure test-fixture-fixup cycle (no new feature work, no contract changes) has no obvious "parent takes it" surface. The 9 pre-existing failures need ROOT-CAUSE analysis (the v0.10.16 D1 deliverable) before any fix lands. Without D1 classification, the parent lacks the D2/D3/D4 site map.
4. **The deferred list was correctly prioritized, but MiMo-availability is the binding constraint.** The 9 failures + the deferred F17/F18 from the v0.10.14 cycle-end audit formed the v0.10.16 brief. The brief is ship-ready (per the M-1 ACK protocol review verdict); the execution is a pure operational question, not a scope question.
5. **Cycle planning documents correctly stay on `main`.** Per project convention, the [`plans/v0.10.16-mimo-half-prompt.md`](./v0.10.16-mimo-half-prompt.md) brief was committed to `main` BEFORE the cycle starts (along with the docs-pack commit `1b9b32c`). This is correct: when MiMo credits return, the brief on `main` is the run-spec. The deferred outcome is a feature, not a bug — the brief is the canonical next-cycle input.
