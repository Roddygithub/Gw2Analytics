## Finding (part 1: alembic `env.py` missing `compare_type` + `compare_server_default`)

`env.py::context.configure()` calls `context.configure(...)`
in 2 places (offline + online mode) WITHOUT
`compare_type=True` or `compare_server_default=True`. The
default is `compare_type=False`, which means alembic's
`--autogenerate` mode does NOT detect column type changes
(e.g., `Integer` ŌåÆ `BigInteger`, `String(50)` ŌåÆ `String(255)`).
The autogenerated migration will silently miss the type
change, and the operator has to hand-edit the migration to
add the `existing_type=` argument to `alter_column()`.

Alembic 1.13+ supports both flags. The flags are documented
in the alembic docs ┬¦"Comparing Metadata":

- `compare_type=True` ŌĆö auto-detect column type changes
  (e.g., `Integer` ŌåÆ `BigInteger`).
- `compare_server_default=True` ŌĆö auto-detect server-side
  default changes (e.g., `'pending'` ŌåÆ `'parsing'` in a
  `server_default=`).

The current migration history (migrations 0001-0008) was
hand-written (no autogenerate), so the missing flags
haven't surfaced yet. The next autogenerated migration
(e.g., a v0.9.x schema addition) will surface them.

## Fix (part 1)

Add the 2 flags to BOTH `context.configure()` call sites
in `env.py`:

```python
context.configure(
    url=url,
    target_metadata=target_metadata,
    literal_binds=True,
    compare_type=True,            # NEW
    compare_server_default=True,  # NEW
)
# ... and:
context.configure(
    connection=connection,
    target_metadata=target_metadata,
    compare_type=True,            # NEW
    compare_server_default=True,  # NEW
)
```

## Finding (part 2: `gw2_api_client` consumed from PyPI, not workspace)

`apps/api/pyproject.toml` line 25 pins `gw2_api_client>=0.1.0`
as a PyPI dep. The root `pyproject.toml` declares
`libs/gw2_api_client` as a workspace member, but the
`[tool.uv.sources]` block has NO `gw2_api_client = { workspace = true }`
line. The result: `uv sync` resolves `gw2_api_client` from
PyPI (the public release), not from the local workspace.

The other 3 libs (gw2_core + gw2_evtc_parser + gw2_analytics)
ARE consumed from the workspace (per the
`[tool.uv.sources]` entries in root `pyproject.toml`).
`gw2_api_client` is the odd one out.

The drift risk: a developer who edits a file under
`libs/gw2_api_client/` and runs `uv run mypy apps/api` sees
the OLD `gw2_api_client` (from PyPI), not their edit. The
edit appears to have no effect until they manually rebuild
the package. The canonical workspace consumption (per the
other 3 libs) avoids this.

## Fix (part 2)

1. Add `gw2_api_client = { workspace = true }` to root
   `pyproject.toml`'s `[tool.uv.sources]` block.
2. Change `apps/api/pyproject.toml` line 25 from
   `"gw2_api_client>=0.1.0"` to `"gw2_api_client"` (no
   version constraint). The workspace source resolves the
   version from the workspace member's pyproject.toml.

The change matches the pattern of the other 3 libs (bare
name, no version constraint, workspace source in root
pyproject.toml).

## Risks

- The `compare_type=True` + `compare_server_default=True`
  flags change the autogenerate behaviour. The next
  autogenerated migration (if any) will include the
  `existing_type=` + `existing_server_default=` arguments
  on `alter_column()` calls. Hand-written migrations (the
  current 0001-0008) are unaffected.
- A future alembic downgrade (if a migration needs to
  revert a column type) requires `postgresql_using=`
  on the `alter_column()` call (Postgres-specific syntax
  for `USING` clauses in ALTER TABLE). The autogenerate
  output will not include this automatically; the operator
  must hand-edit.
- The `gw2_api_client` workspace switch changes the
  resolution: a developer who installed `gw2_api_client`
  from PyPI globally (e.g., via `pip install`) will
  continue to have the global version; `uv sync` will
  NOT override it. The workspace source is honored
  INSIDE the workspace venv only.
- The `apps/api` Docker build (per plan 026) uses
  `uv sync --frozen`; the workspace source is resolved
  at build time and the local `libs/gw2_api_client` is
  copied into the build. The Docker image size is
  unchanged.

## Rejected alternatives

- **Add `compare_type=True` only (not
  `compare_server_default`)**: tempting (the current
  migrations don't have `server_default=`).
  `compare_server_default=True` is needed for future
  migrations that add a `server_default=` (e.g., the
  CHECK constraints in plan 029 have
  `server_default='pending'`). Adding both flags now
  is the v0.9.19 minimum.
- **Switch `gw2_api_client` to a path dep** (`gw2_api_client = { path = "libs/gw2_api_client" }`):
  out of scope. The `path` dep is for non-workspace
  packages; the `workspace = true` source is the
  canonical pattern for workspace members.
- **Pin `gw2_api_client` to an exact version
  (`==0.1.0`)** instead of the bare name: out of scope.
  The workspace source resolves the version from the
  member's pyproject.toml; an exact pin is redundant.
- **Move `gw2_api_client` to a private index** (e.g.,
  a self-hosted devpi): out of scope. The workspace
  source is the v0.9.x canonical pattern.
- **Drop the `compare_server_default` flag and rely
  on the operator's manual review**: out of scope. The
  flag is the canonical auto-detection mechanism per
  the alembic docs.
