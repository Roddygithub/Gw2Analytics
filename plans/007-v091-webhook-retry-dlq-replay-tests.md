## Context

The v0.9.1 webhook slice ships a polling scheduler (`webhook_scheduler.py`) with
1s/10s/100s exponential backoff and DLQ promotion after 3 failed attempts, plus a
replay endpoint that creates a fresh delivery row from a DLQ entry. The only tests
in `apps/api/tests/test_webhooks_e2e.py` cover the v0.9.0 CRUD surface
(create/list/get/revoke + 422/404/204 paths). Race conditions in the scheduler,
concurrent replay idempotency, the HMAC byte-for-byte round-trip across retries, and
the subscription-missing-at-retry edge cases are all unknown â€” they will be found
in production. The `pytest.ini` is bare; `pytest-time-machine` (or `freezegun`) is
not configured despite the time-sensitive workload.

## Steps

1. **Add the dev dependencies**.
   - Verify command: `cd apps/api && uv add --dev pytest-time-machine respx`
   - Expected: pyproject.toml has new entries; `uv lock` regenerates the lockfile.

2. **Write `test_retry_scheduler_first_attempt_success`**.
   - Seed via ORM inserts (existing `_post_sub` helper builds the subscription + secret):
     `OrmWebhookSubscription` + 1 `OrmWebhookDelivery` row with `attempt=1`, `next_attempt_at=now`, `payload={"kind":"upload_completed", ...}`, `status_code=NULL`.
   - Mock `respx.post(...)` to return 200.
   - Call `process_scheduled_retries(session_factory)`.
   - Assert: returns `1`, `delivery.delivered_at` set, `delivery.error is None`, no `OrmWebhookDlq` row created.

3. **Write `test_retry_scheduler_failure_promotes_to_dlq_after_3_attempts`**.
   - Seed identical to step 2.
   - Mock `respx.post(...)` to return 500.
   - Use `time_machine.travel("+1.5s")` between scheduler ticks; repeat 2 more times.
   - Assert: after the third tick, the delivery row is gone and an `OrmWebhookDlq` row exists with `last_error` containing `"non-2xx response: 500"`.

4. **Write `test_replay_dlq_idempotent_concurrent_calls`**.
   - Seed: `OrmWebhookDlq` + corresponding subscription + completed upload (existing `_make_minimal_zevtc` + parse).
   - Spawn 5 threads, each calling `replay_dlq_delivery(delivery_id, db_session)`.
   - Assert: exactly 1 thread sees `HTTP 201`, the other 4 see 404. The DLQ row is absent at the end (atomic delete).

5. **Write `test_replayed_delivery_byte_for_byte_hmac_matches_original`**.
   - Seed: a failed delivery with its `payload={"kind":"upload_completed","upload_id":...,"fight_id":...,"sha256":...,"started_at":...}` (canonical v0.9.0 webhook body shape).
   - Use `hmac.new(sub.secret.encode(), json.dumps(payload, separators=(",",":")).encode(), hashlib.sha256).hexdigest()` to compute the canonical HMAC.
   - Call `replay_dlq_delivery(...)`.
   - Assert: the new delivery row\'s `payload` field round-trips through `json.dumps(payload, separators=(",",":"))` to the same bytes; HMAC re-computed on those bytes matches the original.

6. **Run ruff + mypy + the new test suite**.
   - Verify command: `cd apps/api && uv run ruff check tests/test_webhooks_e2e.py && uv run mypy tests/test_webhooks_e2e.py && uv run pytest tests/test_webhooks_e2e.py -v`
   - Expected: all PASS, plus the existing v0.9.0 CRUD tests still PASS (no regression).

## Maintenance note

The scheduler-poll tests are the canonical reference for "what happens when the worker
ticks." Any future change to `webhook_scheduler.py` (e.g. v0.10 retry-budget
extension, v0.10 fix-after-N-streak retry, v0.10 dead-letter archival) MUST update
these tests. Add a comment at the top of `test_webhooks_e2e.py` linking the
scheduler code path to these test names + their fixtures.

## Escape hatches

- **STOP** if `respx` doesn\'t hook into `httpx.Client` (we use sync `httpx.Client`, not `httpx.AsyncClient`; `respx.sync` should handle sync). Fall back to monkey-patching `httpx.Client.post` via a fixture.
- **STOP** if time-machine advancement conflicts with Postgres timestamp comparison (the `OrmWebhookDelivery.next_attempt_at` is compared via `<= now()`). Use UTC consistently and document.
- **STOP** if the concurrent-replay test reveals a non-atomic promotion path (the existing `db.delete(dlq) + db.add(new_delivery) + db.commit()` is supposed to be atomic â€” verify; if not, escalate to a separate concurrency-correctness plan).
- **STOP** if the HMAC round-trip test reveals that the `payload` stored in `OrmWebhookDelivery.payload` (JSONB) is NOT byte-for-byte equivalent to the original `body_bytes` (e.g. Postgres strips trailing whitespace or re-orders keys after JSONB round-trip). The fix is to store the canonical `bytes` value, not the parsed `dict` â€” separately scoped.
