# 004-v091-webhook-schema-id-type-fix

**Status**: DONE (shipped in v0.9.1 hardening slice)
**Date**: 2026-07-08 (executed as part of H1 + H2 followups)
**Drift-detection base**: `ef5e4f3`
**Addresses finding**: #1 — schema `int` ↔ `str` mismatch on `WebhookDeliveryOut` + `WebhookDeliveryReplayOut`

## Context

The v0.9.1 webhook retry+DLQ+replay slice ships two new Pydantic response schemas
(`WebhookDeliveryOut`, `WebhookDeliveryReplayOut`) that incorrectly type their primary
identifier as `int`. The actual on-disk value, generated by
`routes/webhooks.py:_generate_delivery_id`, is `f"dly_{uuid.uuid4()}"` — a string.
Pydantic raises `ValidationError` the first time any
`/api/v1/webhooks/dlq/{delivery_id}/replay` POST or any GET endpoint that exposes
the broken schemas serializes a `OrmWebhookDelivery` instance. The whole v0.9.1
surface sits behind a hard runtime failure.

The existing v0.9.0 e2e test suite worked around the missing type by
`cast("Response", client.post(...))` in the `_post_sub` helper — a workaround that
masks the response-shape error from the test runner but does NOT mask it from
FastAPI\'s runtime serializer. Once a real consumer (not a test client) hits the
endpoint, they see `500 Internal Server Error` with a Pydantic stacktrace.

## Maintenance note

Any future webhook response schema (v0.9.1++ hardening — e.g. listing DLQ endpoints,
audit views of delivery history) MUST use `str` for IDs generated via
`_generate_delivery_id`. Add a one-line docstring to `_generate_delivery_id` pinning
the discriminator prefix `"dly_"` to the contract as part of this same diff.

## Escape hatches

- **STOP** if the SQL column type for `webhook_deliveries.id` is NOT `String(64)` (verify via `head -120 apps/api/alembic/versions/0006_webhooks.py`). Resolving that requires a new alembic migration — different scope, separate plan.
- **STOP** if mypy complains about a downstream caller that assumed `int` for the id (e.g. an unrelated aggregation join). Report the caller location rather than coerce.
