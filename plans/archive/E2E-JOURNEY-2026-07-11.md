# E2E user-journey validation — findings (2026-07-11)

Full-autonomy real-stack E2E of Gw2Analytics @ `v0.10.25`. Harness:
[`web/e2e/user-journey.spec.ts`](../web/e2e/user-journey.spec.ts) (+ README).

## Context

- **Stack booted OK.** Postgres 16 (`brew postgresql@16`) + MinIO (`brew minio`, S3 blob store) + FastAPI (`:8000`, `ALLOW_INREQUEST_PARSE_FALLBACK=1` → parse in-request, no Redis/worker) + Next.js dev (`:3000`, pointed at the real API). *(Two necessary deviations from a Postgres-only recipe: MinIO is required for the upload blob store; Redis/worker bypassed via the in-request fallback.)* App DB driver is sync `psycopg`, not `asyncpg`.
- **Files tested** (no user `.zevtc` found on the machine — used repo fixtures): `small.zevtc` 5 KB (real 47-agent/2000-skill fight = `web/tests/fixtures/zevtc/real_small.evtc` zipped), `medium.zevtc` 2.4 MB (same fight + padding), `large.zevtc` 101 MB (synthetic, cap test).
- **Pages visited:** landing, `/upload` ×3, `/fights`, `/fights/[id]`, `/players`, `/players/compare`. 20 screenshots captured.

## Screenshots analysed (visual review)

- **Landing** — dark hero, nav (Players/Compare + account search), 4 feature cards, footer wired to the real API. Clean.
- **Small upload** — 3-step wizard works flawlessly: valid-file gate, then jumps to **step 4 "Done", `completed`** (in-request parse is near-instant) with ID/SHA + `/fights/{id}` drill link.
- **Fights list** — AG Grid renders 1 row (enc 1, 47 agents, build 20251115), sortable/filterable columns, pagination.
- **Fight detail** — the whole page is **"Upstream error: 500"** (all six section shots byte-identical). Renders *fast* (timeline fix) but one failing endpoint blanks the entire page.
- **Players** — clean **empty state** ("0 players"); Compare page renders its "add 2 accounts" guidance.
- **Medium upload** — upload step shows **"Upstream error: 500"** (the upload itself failed).
- **Large upload** — **graceful client cap**: "File is too large (101.0 MiB). Maximum is 100.0 MiB.", Next disabled.
- **Positives:** upload wizard, dual-layer 100 MiB cap (client + server 413), empty states, landing/list all correct; no uncaught page errors.

## Bugs found (7)

| # | Severity | Description |
|---|---|---|
| 1 | **HIGH** | `/fights/{id}/timeline` **hangs indefinitely** on non-normalized `time_ms` — `PerFightTimelineAggregator` lacked the `_MAX_BUCKETS` guard `EventWindowAggregator` has → ~2.9e15-bucket alloc → DoS (pinned the worker + hung the detail SSR 30s+). |
| 2 | **HIGH** (parser, root) | `real_small.evtc` **mis-parsed** → garbage `time_ms≈1.4e19` + 0 player summaries (skill-table-misread cascade). Root trigger of #1, of `/events` 500, and of the empty players list. |
| 3 | **MED** | **Upload 500 on `fight_id` collision** — two uploads with the same inner fight (diff outer-zip sha256) → duplicate `fights_pkey` → unhandled `IntegrityError` → 500. |
| 4 | **MED** (UX) | **No per-section error isolation** on `/fights/[id]` — one 500 endpoint blanks the whole page; sections returning 200 aren't shown. |
| 5 | **MED** (CI) | **origin/main CI red** (pre-existing) — `F811` from two `BarrierEvent` class defs in `gw2_core/models.py:542` & `:648` (WAVE-8 merge dup); dies at `Ruff check`, so the whole gate never runs. |
| 6 | **LOW** (perf) | `/timeline/players` ~10s for a 47-agent fight. |
| 7 | **LOW** (frontend) | Hydration mismatch on `PlayerSearchBar` (server longhand CSS + `caret-color:transparent` vs client shorthand). |

*Note:* `/events` returning 500 is **not** a separate bug — its `_MAX_BUCKETS` guard is working correctly on the garbage `time_ms`; the culprit is the parser (#2).

## Bugs fixed

- **#1 timeline hang → `edacc4b`** (plan 159): added `_MAX_BUCKETS=50_000` to `PerFightTimelineAggregator` (parity with `EventWindowAggregator`) + a regression test. **Verified live**: `/timeline` 500 in 0.04s (was ∞), web `/fights/[id]` 200 in 0.45s (was 30s+ hang).

## UX recommendations (prioritized)

1. **Isolate detail-page sections** (#4) — per-section error/empty states so one bad endpoint doesn't blank the page. Biggest perceived-quality win.
2. **Handle `fight_id` collisions** (#3) — return the existing fight (idempotent) or a clear 409, never a 500.
3. **Humanise the detail error** — "This fight couldn't be aggregated (corrupt timing data)" beats a raw "Upstream error: 500".
4. **Lazy-load / speed up `/timeline/players`** (#6) so the detail SSR isn't gated on a ~10s call.
5. **Keep** the upload wizard, dual-layer cap, and empty states — genuinely well done.

## Next step recommended

**Dedupe `BarrierEvent` in `gw2_core/models.py` (#5) to un-red CI** — it blocks the entire lint→test gate, so nothing (incl. the plan-159 fix) is validated on merge. ~2-line fix but it's Roddy's WAVE-8 call on which definition survives + reconciling the statechange dispatch wiring. Then the parser `time_ms` normalization (#2) is the highest-value follow-up — it's the root cause that lets real fights render at all; the timeline guard only stops the hang.
