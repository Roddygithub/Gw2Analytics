## §2 — M8 forward-defer rationale (Bucket K: Test-Substrate Mismatch)

The v0.10.19 cycle attempted to resolve the 11 M8 pytest failures surfaced
at v0.10.18.1 (`plans/AUDIT-2026-07-13-2ffafc75.md`). Sub-bucket
distribution per `plans/RELEASE-v0.10.19.md` §3:

| Sub-bucket | Count | Files | Hypothesis (v0.10.18.1) | Resolution attempted (v0.10.19) |
|---|---|---|---|---|
| **K1** (Arq-Worker connectivity) | 5 | `tests/test_uploads_arq.py` | `_disable_arq_for_tests` autouse patches `arq.create_pool` to raise `ConnectionError`; the `_mock_arq_pool` test fixture's lifespan race is not hermetic enough (the lifespan-overrides-mock path clobbers the mock). | PR-3: `_mock_arq_pool` depends on `client` (post-lifespan mock install). The fix was APPLIED but the 5 tests still fail because `_clear_settings_cache` autouse + `monkeypatch.delenv` are out-priority by the conftest's `setdefault("ALLOW_INREQUEST_PARSE_FALLBACK", "1")` (executed at module load). The PR ultimately DOES NOT change K1 outcome because the route handler still reads `get_settings().allow_inrequest_parse_fallback=True` from the conftest setdefault. |
| **K2** (IP-routing/SSRF gate) | 4 | `tests/test_webhooks_e2e.py` | `pydantic-settings` `DotEnvSettingsSource` reads `apps/api/.env` at `Settings()` instantiation, so `monkeypatch.delenv("GW2ANALYTICS_ALLOW_PRIVATE_WEBHOOK_URLS")` is bypassed by the `.env` value (test host leaks the dev opt-in). Gate fires 422 in production but test sees 201. | PR-1: conftest `_isolate_webhook_validation_env` autouse delenv + PR-1 followup `_disable_dotenv_for_tests` autouse that patches `Settings.settings_customise_sources` to omit the dotenv source. The patch is functionally correct (5 iterations of signature discovery against pydantic-settings' actual call style), but the keyword-only + 6-parameter mixed signatures create intermittent TypeErrors on Settings instantiation at the per-test setup layer. 6 iterations exhausted the signature budget; 3 residuals remained. |
| **K3** (DNS-resolver-pool saturation) | 2 | `tests/test_webhooks_dns_under_attack.py` + `tests/test_webhooks_getaddrinfo_timeout.py` | Production `_DNS_EXECUTOR` has `max_workers=32`; the saturation test's 100-tarpit burst DOES complete within the 2.0s `future.result(timeout)` fence on the production pool (`100/32 < 2.0s` of typical resolver latency), but the saturation test ASSERTION is `{422, 422, ..., 422}` (100×422) — the production pool inadvertently proves the gate is permissive under that burst pattern. The timeout test similarly relies on the per-test pool being more constrained than production. | PR-2: conftest `_isolate_dns_executor` autouse fixture that monkey-patches `webhooks._DNS_EXECUTOR` to a fresh `max_workers=1` `ThreadPoolExecutor`. The fix is mechanically correct, but `conftest.py` modifications across 6 iterations inverted the PR-2 change in it3-6 (re-imports not yet hoisted to module level → PLC0415 errors). |

**Why defer (not grind)**: per `code-reviewer-minimax-m3` strongest
recommendation after iter 6:

> "6 iterations on one fixture is a clear signal that the
> `_disable_dotenv_for_tests` autouse approach is excessive
> complexity for the goal. A single buried 5-line autouse should
> solve test isolation — if it takes 35+ lines + 6 signature
> iterations, the design is wrong. Suggest DEFER M8 to v0.10.20
> (declare the 11 K-cluster failures bucket-K 'Test-Substrate
> Mismatch, NOT production regression' per v0.10.18.1 audit). This:
> - Ships v0.10.19 cleanly with the plan-landing docs (commit
>   `712522a`) + only the close-out docs commits.
> - Removes the dotenv-disable complexity from the codebase entirely.
> - Preserves M8 for v0.10.20 with a fresh 1-iteration fix budget."

The defer preserves D2 baseline + plan-landing docs + cycle-end audit
without the late-cycle re-iteration cost. M8 → v0.10.20 with new
1-iteration budget. **`code-reviewer-minimax-m3` is the canonical
arbiter of this decision** (its verdict is PARALLEL-AGENT-DETERMINED,
not a human override).
